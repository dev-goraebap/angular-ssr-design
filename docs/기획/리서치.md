# Angular SSR / CSR 혼합 전략 리서치

> 작성일: 2026-06-22
> 목적: Angular 환경에서 SSR과 CSR을 우아하게 혼합하기 위한 사전 조사
> 기준 버전: Angular 19에서 하이브리드 렌더링 도입 → 21에서 서버 라우팅 / 증분 하이드레이션 안정화 → **22에서 증분 하이드레이션이 기본값(on-by-default)** 으로 승격. 최신 22 기준으로 정리

> 버전 흐름 요약: **19**(하이브리드 렌더링·서버 라우팅 도입, 증분 하이드레이션 experimental) → **20~21**(서버 라우팅/증분 하이드레이션 stable) → **22**(증분 하이드레이션 기본 활성화, `resource`/`httpResource` + Transfer State 통합, Signal Forms stable, OnPush 기본화). 자세한 22 변화는 [§5](#5-angular-22에서-달라진-점-ssr-관점) 참고.

## 0. 한눈에 보는 결론

- Angular에는 Next.js의 **Server Components / Server Actions / API Routes 처럼 "서버에서만 실행되는 별도 실행 모델"이 없다.** 같은 컴포넌트 코드가 서버에서 한 번(SSR) 실행되고, 브라우저에서 다시 살아난다(hydration). "서버 전용 코드"라는 1급 개념은 없고, **플랫폼 분기(provider 교체 / 라이프사이클 훅)**로 흉내 낸다.
- SSR을 켜더라도 **모든 페이지가 무조건 SSR이 되는 것은 아니다.** Angular 19+의 **서버 라우팅(`app.routes.server.ts`)**에서 라우트 단위로 `RenderMode.Server`(SSR) / `RenderMode.Client`(CSR) / `RenderMode.Prerender`(SSG)를 골라 섞을 수 있다. 컴포넌트 단위 CSR화는 `@defer` + 증분 하이드레이션으로 처리한다.
- 서버↔백엔드 호출은 **서버 간 통신(CORS 무관, 내부 호스트/프록시 사용 가능)**, 브라우저 호출은 **CORS 적용 대상**. 이 간극은 ① `withHttpTransferCache()`로 중복 호출 제거, ② 환경/인터셉터로 baseURL·인증 토큰을 플랫폼별로 주입, ③ 인증은 **localStorage 대신 쿠키 기반으로 통일**하는 패턴으로 메운다.

## 1. Angular에는 "서버 전용으로만 도는 기술"이 없는가?

### 핵심: 메타프레임워크와의 모델 차이

Next.js(App Router)는 **React Server Components(RSC)**, **Server Actions**, **Route Handlers(API Routes)** 처럼 "브라우저로 코드가 절대 가지 않고 서버에서만 실행되는" 1급 개념을 제공한다.

**Angular에는 이에 1:1로 대응하는 "서버 전용 실행 단위"가 없다.** Angular의 SSR 모델은 본질적으로 다음과 같다:

- **같은 컴포넌트 트리**가 서버(Node/Express 등)에서 한 번 렌더되어 완성된 HTML을 내려보내고,
- 브라우저에서 같은 앱이 부팅되어 그 HTML을 **하이드레이션**(이벤트 바인딩을 다시 붙여 살아나게 함)한다.

즉 컴포넌트는 "서버용/클라이언트용"이 나뉘는 게 아니라, **양쪽에서 모두 실행되는 것을 전제로** 작성된다. 공식 문서도 "Angular는 명시적인 '서버 전용 코드' 영역을 지정하지 않는다(Angular doesn't designate explicit 'server-only' code sections)"고 명시한다.

### 그러면 "서버에서만" 동작시키고 싶을 땐?

진짜 서버 전용 로직(파일시스템 접근, 내부 DB 직결, 시크릿 사용 등)이 필요하면 **Angular 앱 안에서가 아니라, SSR을 호스팅하는 Node 서버(`server.ts` / Express 핸들러) 레벨**에서 처리하는 것이 정석이다. Angular는 "프레임워크가 서버 코드를 대신 써주는" 영역이 얇고, 그 부분은 개발자가 직접 Express/Hono 등으로 구성한다.

Angular 앱 코드 내부에서 플랫폼을 가르는 도구는 다음과 같다:

| 도구 | 용도 |
|------|------|
| `isPlatformServer()` / `isPlatformBrowser()` | 런타임 플랫폼 분기 (남용 비권장) |
| `afterNextRender()` / `afterEveryRender()` | **브라우저에서만** 실행되는 라이프사이클 훅 (DOM 접근 등) |
| 플랫폼별 provider 교체 | 같은 인터페이스를 서버용/클라용 구현으로 DI 주입 (권장) |
| `REQUEST` / `RESPONSE_INIT` 인젝션 토큰 | 서버에서 현재 요청 객체 / 응답 헤더·상태코드 접근 |

> 공식 문서 권장사항: 템플릿 안에서 `isPlatformBrowser` 분기를 흩뿌리지 말고, **"플랫폼별 provider"로 동작을 갈아끼우라**("Prefer platform-specific providers over runtime checks"). 즉 `WINDOW`, `STORAGE` 같은 토큰을 만들어 서버에선 no-op/모킹 구현을, 브라우저에선 실제 구현을 주입하는 방식.

### 요약

- **질문(서버에서만 도는 기술이 없는가?) → 사실상 맞다.** Angular는 "서버 전용 실행 모델"을 1급으로 제공하지 않는다.
- 서버 전용 작업은 (a) SSR 호스트 서버(Express) 레벨에서 하거나, (b) 앱 내부에서 플랫폼 분기/provider 교체로 흉내 낸다.
- 단, 아래에서 다룰 **`@defer` + `hydrate never`** 처럼 "이 컴포넌트는 서버에서 HTML만 그리고 브라우저 JS는 보내지 않기" 같은 부분적 통제는 가능하다.

## 2. SSR을 켜면 모든 페이지가 SSR인가? 특정 페이지/컴포넌트만 CSR 가능?

### 페이지(라우트) 단위 — 가능하다 (하이브리드 렌더링 / 서버 라우팅)

Angular 19부터 **서버 라우팅(server routing)** 으로 라우트마다 렌더링 모드를 지정할 수 있다. 보통 `app.routes.server.ts`에 `ServerRoute[]` 배열을 만들고, `provideServerRendering(withRoutes(serverRoutes))`로 등록한다.

세 가지 모드:

| RenderMode | 의미 | 적합한 페이지 |
|------------|------|---------------|
| `RenderMode.Server` | **요청마다 서버에서 HTML 렌더(SSR)** | 동적/개인화/인증 필요 페이지 |
| `RenderMode.Client` | **브라우저에서만 렌더(CSR)** — Angular 기본 동작 | 대시보드, 로그인 후 영역, SEO 불필요·고도 인터랙션 페이지 |
| `RenderMode.Prerender` | **빌드 타임에 정적 HTML 생성(SSG)** | 마케팅/블로그/거의 안 변하는 콘텐츠 |

예시:

```ts
// app.routes.server.ts
import { RenderMode, ServerRoute } from '@angular/ssr';

export const serverRoutes: ServerRoute[] = [
  { path: '',            renderMode: RenderMode.Prerender }, // 랜딩 → SSG
  { path: 'blog/:slug',  renderMode: RenderMode.Server },    // 글 상세 → SSR
  { path: 'dashboard',   renderMode: RenderMode.Client },    // 대시보드 → CSR
  { path: '**',          renderMode: RenderMode.Server },
];
```

```ts
// app.config.server.ts
import { provideServerRendering, withRoutes } from '@angular/ssr';
import { serverRoutes } from './app.routes.server';

export const serverConfig = {
  providers: [provideServerRendering(withRoutes(serverRoutes))],
};
```

**즉 "SSR을 켰다 = 전 페이지 SSR"이 아니다.** 라우트별로 SSR / CSR / SSG를 자유롭게 섞는 것이 Angular의 권장 하이브리드 모델이다.

#### Prerender의 동적 파라미터 & 폴백

- `RenderMode.Prerender` 라우트는 `getPrerenderParams()`로 어떤 파라미터 조합을 미리 생성할지 지정한다. 이 함수 안에서 `inject()`로 서비스를 받아 API 호출 → 생성 대상 목록을 만들 수 있다.
- 미리 생성되지 않은 경로에 대한 폴백: `PrerenderFallback.Server`(요청 시 SSR), `PrerenderFallback.Client`(CSR), `PrerenderFallback.None`.

### 컴포넌트 단위 — `@defer` + 증분 하이드레이션(Incremental Hydration)

"페이지는 SSR인데 특정 무거운 컴포넌트만 나중에/클라이언트에서"를 다루는 것이 **증분 하이드레이션**이다(Angular 19 도입, 21에서 안정화).

핵심 동작:

- `@defer` 블록에 **`hydrate` 트리거**를 붙이면, Angular는 그 블록의 의존성을 **SSR 시점에 로드해 `@placeholder`가 아닌 실제 템플릿을 HTML로 렌더**한다.
- 사용자는 HTML이 이미 있으니 콘텐츠를 즉시 본다. 다만 해당 컴포넌트는 트리거 전까지 **"탈수(dehydrated)" 상태 = 정적 HTML 껍데기**로 남고, JS는 나중에 받는다.
- 트리거가 발동하면 그제서야 컴포넌트 코드를 받아 인터랙티브해진다.

트리거 종류:

| 트리거 | 의미 |
|--------|------|
| `hydrate on <trigger>` | viewport 진입, idle, interaction, hover 등 조건에서 하이드레이션 |
| `hydrate when <expr>` | 표현식이 참이 될 때 하이드레이션 |
| `hydrate never` | **절대 하이드레이션하지 않음** → 서버가 그린 HTML만 남고 그 컴포넌트의 JS는 클라이언트로 가지 않음(정적 콘텐츠) |

```html
@defer (hydrate on viewport) {
  <app-heavy-chart [data]="data()" />
} @placeholder {
  <div class="skeleton"></div>
}
```

> `hydrate never`는 "이 영역은 서버 렌더 결과만 쓰고 클라 JS 비용 0"이라는, Next.js의 서버 컴포넌트와 가장 유사한 효과를 내는 도구다(완전히 동일하진 않음).

### 전제: 하이드레이션 활성화 (버전별 차이 주의)

위 모든 것은 하이드레이션이 켜져 있어야 한다.

- **Angular 19~21**: `provideClientHydration(withIncrementalHydration())`(+ `withEventReplay()`)처럼 증분 하이드레이션을 **명시적으로 켜야** 했다.
- **Angular 22+**: **`provideClientHydration()`만 호출해도 증분 하이드레이션이 기본 활성화**된다. 끄고 싶으면 `withNoIncrementalHydration()`으로 명시적 opt-out 한다.

증분 하이드레이션은 전체 하이드레이션 + deferrable views + event replay 위에 쌓인 기능이다.

### 요약

- **페이지 단위**: `app.routes.server.ts`에서 라우트별 `RenderMode`로 SSR/CSR/SSG 혼합. → "특정 페이지만 CSR" 쉽게 가능.
- **컴포넌트 단위**: `@defer (hydrate ...)`로 부분 하이드레이션, `hydrate never`로 사실상 클라 JS 제거. → "특정 컴포넌트만 다르게" 가능.

## 3. SSR/CSR 간 백엔드 API 호출 간극을 우아하게 다루기

질문에 담긴 세 가지 쟁점을 각각 정리한다.

### 3-1. 서버(SSR)에서 백엔드를 호출할 때 — CORS / 내부 호스트 / 프록시

**맞다. 서버→서버 호출에는 CORS가 없다.** CORS는 브라우저가 강제하는 보안 메커니즘이므로, SSR Node 서버가 백엔드를 호출하는 것은 일반 HTTP 클라이언트 호출일 뿐 CORS 프리플라이트/Origin 검사 대상이 아니다.

따라서 SSR 단계에서는:

- **내부 호스트(`http://api-internal:8080`)나 같은 클러스터의 서비스 디스커버리 주소, nginx 프록시(`http://localhost/api`)로 호출 가능.** HTTPS도 필수가 아니다(사설망 내부 통신).
- 외부 공개 도메인(`https://api.example.com`)으로 나갈 필요가 없어 **레이턴시·홉 수를 줄일 수 있다.**

다만 주의:

- SSR에서 호출하는 baseURL과 브라우저에서 호출하는 baseURL이 **서로 다르다**(내부 호스트 vs 공개 도메인). 이걸 환경/플랫폼별로 깔끔히 분리해야 한다(아래 3-2).
- 상대경로(`/api/...`)를 그대로 쓰면 브라우저에선 현재 origin 기준으로 잘 동작하지만, **서버에는 "현재 origin"이 없다.** SSR HttpClient는 절대 URL을 요구하므로, 서버 측에서는 baseURL을 붙여 절대경로로 만들어줘야 한다.

### 3-2. 브라우저(CSR)에서 호출할 때 — CORS / 환경별 baseURL 관리

- 브라우저 `fetch`/`HttpClient`는 **내부 호스트로 직접 호출 불가**(사용자 망에서 사설 호스트가 안 보임). 공개 엔드포인트(보통 HTTPS)로 가야 한다.
- 백엔드와 프론트 origin이 다르면 **CORS 설정 필요**. 쿠키 인증을 쓰면 `withCredentials: true`(fetch는 `credentials: 'include'`) + 서버의 `Access-Control-Allow-Credentials: true`가 필요하고, 이 경우 `Access-Control-Allow-Origin: *`(와일드카드)는 쓸 수 없다(구체 origin 명시 필요).

**깔끔한 관리 전략 (권장 순):**

1. **같은 origin으로 합치기 (가장 깔끔, CORS 자체를 없앰)**
   - 운영: 프론트(SSR 서버)와 백엔드를 같은 도메인 뒤에 두고 nginx 등 리버스 프록시로 `/api`를 백엔드로 라우팅. 브라우저는 `/api/...` 상대경로만 호출 → **CORS 없음**.
   - 개발: `proxy.conf.json`(`ng serve --proxy-config`)으로 `/api`를 백엔드 dev 서버로 프록시 → 개발에서도 CORS 없음.

2. **baseURL을 플랫폼/환경별로 주입**
   - `API_BASE_URL` 같은 InjectionToken을 두고, 서버 설정(`app.config.server.ts`)에서는 내부 호스트, 클라 설정에서는 공개 도메인(또는 빈 문자열=상대경로)을 provide.
   - 환경 차이(dev/stage/prod)는 `environment.ts` 파일 분리 또는 런타임 환경 주입으로 관리.

3. **HttpInterceptor에서 baseURL/절대경로 보정**
   - 인터셉터에서 `isPlatformServer()`이면 상대경로 앞에 서버용 baseURL을 붙이고, 브라우저에선 그대로 두는 방식으로 한 곳에서 처리.

```ts
export const API_BASE_URL = new InjectionToken<string>('API_BASE_URL');

// app.config.server.ts → { provide: API_BASE_URL, useValue: 'http://api-internal:8080' }
// app.config.ts(클라)  → { provide: API_BASE_URL, useValue: '' } // 상대경로 = 같은 origin

export const baseUrlInterceptor: HttpInterceptorFn = (req, next) => {
  const base = inject(API_BASE_URL);
  const url = req.url.startsWith('http') ? req.url : `${base}${req.url}`;
  return next(req.clone({ url }));
};
```

### 3-3. 중복 호출 방지 — `withHttpTransferCache()` (TransferState)

SSR이 서버에서 데이터를 받아 HTML을 그렸는데, 브라우저가 하이드레이션하며 **같은 API를 또 호출**하면 낭비다. Angular는 이를 위한 내장 메커니즘을 제공한다.

- `provideClientHydration(withHttpTransferCache())` (또는 `provideClientHydration()` 기본 포함)를 켜면, **서버에서 발생한 HttpClient 요청 결과를 직렬화해 초기 HTML에 실어 보내고**, 브라우저는 초기 렌더 시 그 캐시를 재사용해 **중복 GET 호출을 건너뛴다.**
- 기본 동작: **`Authorization`/`Proxy-Authorization` 헤더가 없는 `GET`/`HEAD` 요청만** 캐싱.
- 효과: 콘텐츠 위주 페이지에서 보통 1~3개의 중복 호출을 제거, 평균 네트워크에서 100~500ms 단축.

**Angular 22 — `resource()` / `httpResource()` 와 Transfer State 통합:**

- 22부터는 신호 기반 비동기 API인 `resource()` / `httpResource()`가 stable이 되었고, **서버에서 미리 로드된 resource가 HTTP Transfer State와 자연스럽게 연동**된다. 즉 서버에서 `httpResource`로 가져온 데이터가 브라우저 초기 렌더에서 그대로 재사용되어 **중복 호출과 "데이터 재로딩 깜빡임"이 사라진다.** 새 코드에서는 수동 `HttpClient` + 인터셉터 조합보다 `httpResource`를 우선 고려할 만하다.

**주의(데이터 누수/오염 위험):**

- 응답이 헤더(인증/테넌트/로케일)에 따라 달라지는 경우 transfer cache를 쓰면 안 된다. 캐시 키가 그 헤더들을 구분하지 못해 **다른 사용자 데이터가 섞일 수 있다.** 쿼리 파라미터 순서만 달라도 별도 키가 되어 중복 호출이 생길 수 있다.
- 인증이 필요한 개인화 응답은 캐시 대상에서 제외하거나, 캐시 옵션을 커스터마이즈한다.

### 3-4. 인증 통일 — localStorage 문제와 쿠키 기반 통일

**문제의 핵심:** CSR에서는 모든 요청이 브라우저에서 시작하므로 쿠키/`localStorage` 토큰이 자동으로 따라간다. 하지만 **SSR에서는 요청이 서버에서 시작**하므로 사용자의 브라우저 저장소에 접근할 수 없다.

- **`localStorage`/`sessionStorage`는 서버에 존재하지 않는다.** SSR 단계에서 `localStorage` 토큰에 의존하면 서버는 토큰을 못 읽어 비로그인 상태로 렌더된다 → 깜빡임/오데이터/하이드레이션 불일치.

**권장: 인증을 쿠키 기반으로 통일**

1. 토큰을 `localStorage`가 아니라 **쿠키(가급적 `HttpOnly` + `Secure` + `SameSite`)**에 저장.
2. 브라우저 요청: 쿠키가 자동 전송(같은 origin이면 자연스럽게, 크로스 origin이면 `withCredentials`).
3. SSR 요청: HttpInterceptor에서 `isPlatformServer()`일 때 **`REQUEST` 토큰으로 들어온 요청의 `Cookie` 헤더를 읽어**, 백엔드로 나가는 HttpClient 요청에 `Cookie`/`Authorization` 헤더로 **수동 포워딩**한다.

```ts
export const authForwardInterceptor: HttpInterceptorFn = (req, next) => {
  const platformId = inject(PLATFORM_ID);
  if (isPlatformServer(platformId)) {
    const request = inject(REQUEST);            // @angular/core
    const cookie = request?.headers.get('cookie');
    if (cookie) {
      req = req.clone({ setHeaders: { cookie } }); // 사용자 쿠키를 백엔드로 전달
    }
  }
  // 브라우저에선 쿠키가 자동 전송되므로 그대로 통과 (withCredentials 설정 시)
  return next(req);
};
```

- 이렇게 하면 **서버 호출과 클라 호출의 인증을 같은 "쿠키" 한 줄기로 통일**할 수 있다. 인터셉터 한 곳에서 플랫폼 분기만 하면 된다.
- 만약 `HttpOnly` 쿠키라 클라 JS가 토큰을 못 읽는 구조라면, 인증 상태 판단은 서버 렌더 결과/전용 엔드포인트(`/me`)에 맡기는 것이 깔끔하다.
- 토큰을 `localStorage`에 둘 수밖에 없는 레거시라면, 최소한 **SSR 단계에서는 인증 의존 영역을 `@defer (hydrate ...)`나 `RenderMode.Client`로 빼서** 서버가 토큰 없이 렌더하는 문제를 회피한다.

### 3-5. 종합 권장 아키텍처

```
[브라우저] --(상대경로 /api, 쿠키 자동)--> [nginx/리버스프록시] --> [백엔드]
                                                ↑ 같은 origin → CORS 없음
[SSR Node 서버] --(내부호스트 http://api-internal, 요청쿠키 포워딩)--> [백엔드]
                                                ↑ 서버간 통신 → CORS 없음
```

- 같은 origin 프록시로 **CORS 문제를 구조적으로 제거**.
- baseURL은 `InjectionToken` + 플랫폼별 provider로 분리.
- 인증은 **쿠키로 통일**하고, SSR에서는 `REQUEST`로 쿠키를 포워딩.
- 중복 호출은 `withHttpTransferCache()`로 제거하되, 개인화 응답은 캐시 제외.

## 5. Angular 22에서 달라진 점 (SSR 관점)

Angular 22(2026년 릴리스)는 21까지의 "실험/옵트인" 기능들을 **기본값으로 승격**시킨 버전이다. SSR/하이드레이션 관점의 핵심 변화:

| 항목 | 21 이하 | 22 |
|------|---------|-----|
| 증분 하이드레이션 | 명시적 `withIncrementalHydration()` 필요 | **`provideClientHydration()`만으로 기본 활성화**, 끄려면 `withNoIncrementalHydration()` |
| `resource()` / `httpResource()` | experimental | **stable**, 서버 프리로드 데이터가 Transfer State와 통합 → 중복 호출/깜빡임 제거 |
| Signal Forms (`@angular/forms/signals`) | experimental | **stable** (프로젝트 컨벤션상 신규 폼은 Signal Forms 우선) |
| 변경 감지 | OnPush는 선택 | **OnPush가 기본**(옛 동작은 `ChangeDetectionStrategy.Eager`로 명명, 마이그레이션 툴 제공) |
| platform-server 보안 | — | **SSRF/경로 하이재킹 방어 강화**: 의심스러운 URL·프로토콜 상대 URL 거부, 백슬래시 URL을 통한 `HttpClient` SSRF 우회 차단 |
| 라우트별 렌더링 | 서버 라우팅으로 SSR/CSR/Prerender | 라우터에서 라우트별 렌더링 전략(SSR/CSR/증분 하이드레이션)을 더 명확히 지정 |
| 테스트 러너 | Karma | **Vitest가 기본** (SSR 직접 관련은 아니나 툴체인 변화) |

**이 프로젝트에 주는 시사점:**

1. **하이드레이션 기본 ON** — 22에서 시작하면 별도 설정 없이 증분 하이드레이션이 작동하므로, "끌지 말지"를 의식적으로 결정(`withNoIncrementalHydration()`)하는 쪽이 된다.
2. **`httpResource` 우선 검토** — SSR 중복 호출 문제(§3-3)를 수동 인터셉터/TransferCache 대신 `httpResource`로 더 적은 코드로 해결할 수 있다.
3. **보안 강화 인지** — SSR 서버에서 내부 호스트/프록시로 호출할 때(§3-1) platform-server의 SSRF 방어가 프로토콜 상대 URL·백슬래시 URL을 거부하므로, **내부 호출 URL을 정상적인 절대 URL(`http://api-internal:8080/...`) 형태로 구성**해야 한다.

> 주의: 22의 세부 동작(특히 보안 방어의 정확한 거부 규칙, 라우터 렌더링 전략 API 시그니처)은 마이너 릴리스에서 다듬어질 수 있으므로, 실제 구현 시 해당 버전의 공식 changelog/마이그레이션 가이드를 한 번 더 확인할 것.

## 6. 이 프로젝트에서 검증해볼 시나리오 (다음 단계 제안)

1. 라우트 3종 구성: 랜딩(`Prerender`), 콘텐츠 상세(`Server`), 로그인 후 대시보드(`Client`)를 한 앱에서 혼합.
2. `@defer (hydrate on viewport)`로 무거운 위젯을 부분 하이드레이션, `hydrate never`로 정적 영역 JS 제거 효과 측정.
3. nginx 프록시 + `API_BASE_URL` 토큰으로 dev/prod baseURL 분리, CORS 없는 구성 재현.
4. 쿠키 기반 인증 + `REQUEST` 포워딩 인터셉터로 SSR/CSR 인증 통일 PoC.
5. `withHttpTransferCache()` on/off로 중복 호출 수·LCP 비교.
