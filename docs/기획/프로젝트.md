# 프로젝트 기획 — 영화 카탈로그 (가제: CineCat)

> 작성일: 2026-06-22
> 범위: **도메인/기능/콘텐츠 기획만** 다룬다. 렌더링 전략(SSR/CSR/SSG)·기술 스택·아키텍처 결정은 `docs/설계/`에서 별도로 다룬다.

---

## 1. 프로젝트 목적

Angular 22 환경에서 **SSR과 CSR을 우아하게 혼합**하는 패턴을 시연하기 위한 예제 애플리케이션이다. 단, 이 문서는 그 시연을 담을 **그릇(도메인)** 을 정의하는 데 집중한다.

도메인으로 **영화 카탈로그**를 선택한 이유:

- **공개 콘텐츠(목록·상세)** 와 **개인화 콘텐츠(위시리스트·내 평점)** 가 자연스럽게 공존한다 → 다양한 렌더링 요구를 한 도메인에서 보여주기 좋다.
- 데이터 모델이 단순하다(영화 1개 엔티티 중심).
- 페이지 수가 적당히 나오면서 사용자 흐름이 직관적이다.
- 누구나 이해하는 친숙한 도메인이라 기술 데모의 본질에 집중할 수 있다.

> 데이터는 외부 API 없이 **자체 목(mock) 데이터**로 구성한다(실제 영화 서비스가 아님).

## 2. 핵심 사용자 & 시나리오

### 사용자 유형

| 유형 | 설명 | 인증 |
|------|------|------|
| **방문자(Guest)** | 로그인 없이 둘러보는 사용자 | 불필요 |
| **회원(Member)** | 로그인 후 위시리스트/평점을 관리하는 사용자 | 필요 |

> 회원 가입 절차는 시연 단순화를 위해 생략하고, **데모 계정 로그인**만 제공한다(예: 아이디/비밀번호 1쌍).

### 대표 시나리오

1. **둘러보기**: 방문자가 홈에 들어와 인기/최신 영화를 보고, 목록에서 마음에 드는 영화 상세로 들어간다.
2. **탐색**: 장르로 필터링하거나 제목으로 검색해 원하는 영화를 찾는다.
3. **찜하기**: 회원이 영화를 위시리스트에 담고, 나중에 위시리스트에서 다시 본다.
4. **평가하기**: 회원이 본 영화에 별점을 남기고, "내 평점" 목록에서 관리한다.

## 3. 도메인 모델

### 3.1 Movie (영화) — 핵심 엔티티

| 필드 | 타입 | 설명 |
|------|------|------|
| `id` | string | 고유 식별자 (URL slug 겸용 가능) |
| `title` | string | 제목 |
| `originalTitle` | string? | 원제 |
| `year` | number | 개봉 연도 |
| `genres` | string[] | 장르 키 목록 (예: `["action", "sci-fi"]`) |
| `runtime` | number | 상영 시간(분) |
| `synopsis` | string | 줄거리 |
| `director` | string | 감독 |
| `cast` | string[] | 주요 출연진 |
| `posterColor` | string | 포스터 대체용 색상(목 데이터라 실제 이미지 대신 컬러 블록 사용) |
| `posterUrl` | string? | (선택) 포스터 이미지 경로 |
| `ratingAverage` | number | 평균 평점 (0~5) |
| `ratingCount` | number | 평점 참여 수 |
| `featured` | boolean | 홈 추천 노출 여부 |

### 3.2 Genre (장르)

| 필드 | 타입 | 설명 |
|------|------|------|
| `key` | string | 식별자 (URL용, 예: `sci-fi`) |
| `label` | string | 표시명 (예: `SF`) |

초기 장르 셋(예): 액션, 드라마, SF, 코미디, 스릴러, 애니메이션, 로맨스, 공포.

### 3.3 User (사용자/세션)

| 필드 | 타입 | 설명 |
|------|------|------|
| `id` | string | 사용자 식별자 |
| `name` | string | 표시 이름 |

세션은 로그인 시 발급되는 토큰으로 식별한다(구체적 인증 방식은 설계 문서에서 결정).

### 3.4 WishlistItem (위시리스트 항목)

| 필드 | 타입 | 설명 |
|------|------|------|
| `userId` | string | 소유자 |
| `movieId` | string | 대상 영화 |
| `addedAt` | string(ISO) | 담은 시각 |

### 3.5 Rating (내 평점)

| 필드 | 타입 | 설명 |
|------|------|------|
| `userId` | string | 평가자 |
| `movieId` | string | 대상 영화 |
| `score` | number | 별점 (1~5) |
| `ratedAt` | string(ISO) | 평가 시각 |

### 관계 요약

```
User 1 ── N WishlistItem N ── 1 Movie
User 1 ── N Rating       N ── 1 Movie
Movie N ── N Genre
```

## 4. 화면(페이지) 구성

> 렌더링 모드(SSR/CSR/SSG) 매핑은 의도적으로 비워 둔다 → `docs/설계/`에서 결정.

| # | 경로 | 화면 | 인증 | 핵심 내용 |
|---|------|------|------|-----------|
| 1 | `/` | **홈/랜딩** | 공개 | 서비스 소개, 추천(featured) 영화, 장르 바로가기 |
| 2 | `/movies` | **영화 목록** | 공개 | 전체 영화 그리드, 정렬(연도/평점), 페이지네이션 또는 무한 스크롤 |
| 3 | `/movies/:id` | **영화 상세** | 공개 | 줄거리·감독·출연·평점, 위시리스트 담기, 별점 남기기 |
| 4 | `/genre/:key` | **장르별 목록** | 공개 | 특정 장르의 영화 목록 |
| 5 | `/search` | **검색 결과** | 공개 | 제목 검색 결과(쿼리 파라미터 `q`) |
| 6 | `/wishlist` | **내 위시리스트** | 회원 | 담은 영화 목록, 제거 |
| 7 | `/me/ratings` | **내 평점** | 회원 | 내가 남긴 별점 목록, 수정/삭제 |
| 8 | `/login` | **로그인** | 공개 | 데모 계정 로그인, 로그인 후 원래 페이지로 복귀 |
| 9 | `/settings` | **설정** | 공개 | 다크/라이트 테마 전환 |

### 글로벌 내비게이션

- 상단 또는 하단(모바일)에 핵심 진입점: **홈 · 영화 · 검색 · (회원) 위시리스트/내 평점 · 로그인/로그아웃**
- 모바일 우선 설계이므로 내비게이션 형태(탭바/사이드 등)는 설계 문서에서 결정.

## 5. 기능 요구사항

### 공개(비회원 포함)
- [ ] 홈에서 추천 영화·장르 진입점 노출
- [ ] 영화 목록 조회 (정렬: 최신순/평점순)
- [ ] 영화 상세 조회 (줄거리, 메타데이터, 평균 평점)
- [ ] 장르별 필터링
- [ ] 제목 검색

### 회원 전용
- [ ] 데모 계정 로그인 / 로그아웃
- [ ] 위시리스트 담기 / 빼기 / 목록 조회
- [ ] 별점 남기기 / 수정 / 삭제 / 내 평점 목록 조회
- [ ] 비로그인 상태에서 회원 기능 시도 시 로그인으로 유도 후 복귀

### 설정
- [ ] 다크/라이트 테마 전환(선택은 영속화)

### 비기능(도메인 관점)
- [ ] 모바일 우선(Mobile-first) 경험
- [ ] 접근성: WCAG AA 준수(포커스 관리, 색 대비, ARIA) — 프로젝트 공통 규칙
- [ ] 공개 페이지는 검색엔진 노출(SEO)을 염두에 둔다 → 구체 전략은 설계에서

## 6. 콘텐츠/데이터 규모(시연 기준)

- 영화: **약 15~20편** (장르가 골고루 분포하도록 구성)
- 장르: **6~8종**
- 데모 계정: **1개** (예: `demo` / `demo1234`)
- 위시리스트/평점: 로그인 후 런타임에 생성(초기엔 비어 있음)

---

## 7. 범위 밖 (Non-Goals)

- 실제 회원가입/비밀번호 재설정/소셜 로그인
- 결제·예매·스트리밍 등 실제 서비스 기능
- 외부 영화 API 연동(목 데이터로 대체)
- 관리자(영화 등록/수정) 화면
- 댓글·리뷰 텍스트(별점만 다룸)
- 다국어(i18n) — 필요 시 추후 별도 논의

---

## 8. 열린 질문 (설계로 넘길 항목)

- 각 페이지를 SSR/CSR/SSG 중 무엇으로 렌더할지
- 목록의 페이지네이션 vs 무한 스크롤
- 모바일 내비게이션 형태(하단 탭바 vs 사이드)
- 인증 저장 방식(쿠키 vs 토큰)과 SSR 인증 처리
- 디자인 시스템/UI 컴포넌트 전략(완성형 프레임워크 vs 헤드리스 + 자체 구성)

→ 위 항목은 `docs/설계/` 문서에서 결정한다.
