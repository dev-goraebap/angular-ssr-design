# ADR-0001 · 모바일-first UI 기술스택

> 상태: ✅ 결정됨
> 일자: 2026-06-22
> 관련: [모바일 웹 UX 디자인 스킬], [`docs/기획/프로젝트.md`](../기획/프로젝트.md), [`docs/기획/리서치.md`](../기획/리서치.md)

---

## 맥락 (Context)

이 프로젝트의 본질은 **Angular 22에서 SSR/CSR을 우아하게 혼합하는 패턴 시연**이다(리서치 문서 참고). UI는 그 시연을 담는 그릇이되, **모바일-first**로 "웹 티"가 아니라 앱처럼 느껴지는 경험을 목표로 한다.

따라서 UI 기술 선택의 평가 기준은 일반적인 "빠른 출발 vs 제어권"에 더해, **이 프로젝트 고유의 제약**이 1순위로 들어온다:

1. **SSR/하이드레이션 호환성** — 선택지가 SSR·`@defer`·증분 하이드레이션 흐름과 충돌하면 안 된다. (가장 중요)
2. **모바일 UX 자력** — 적응형 시트(바텀시트↔모달), 끌어서 닫기, 스와이프, safe-area, 44px 터치 타깃.
3. **제어권 / 디자인 소유** — 데모 코드가 프레임워크 추상화에 가려지지 않고 투명해야 한다.
4. **시연 본질과의 정합** — UI가 SSR 데모를 가리지 않아야 한다.

### 모바일 UX 방법론의 전제

참조하는 "모바일 웹 UX 디자인" 방법론은 **"외형은 소유하고 동작은 빌린다"** 를 핵심으로 한다. 즉 포커스 트랩·오버레이·제스처 같은 헤드리스 동작은 검증된 라이브러리에서 차용하고, 그 위에 배선과 스타일만 얹어 **도메인을 모르는 공용 컴포넌트**가 헤드리스 동작과 모바일 UX를 안으로 품게 한다.

## 검토한 대안 (Options Considered)

| 선택지 | SSR 적합성 | 모바일 UX 자력 | 제어권 | 시연 정합 |
|--------|-----------|----------------|--------|-----------|
| Ionic | △ 웹컴포넌트라 하이드레이션·`@defer`와 결이 어긋남, 무거움 | ◎ | ✕ Ionic처럼 보임 | △ 추상화가 데모를 가림 |
| Angular Material | ○ SSR 지원하나 번들 큼, 모바일 제스처/시트 약함 | △ | △ | ○ |
| **spartan/ui (Brain+Helm)** ✅ | ○ Angular 네이티브 헤드리스(CDK 기반), SSR 무난 | ○ (헤드리스 + 제스처는 별도) | ○ Helm을 프로젝트로 복사해 소유 | ○ |
| 헤드리스 완전 소유 (CDK + Tailwind + GSAP) | ◎ | ○ | ◎ | ◎ |

---

## 결정 (Decision)

**UI 라이브러리는 spartan/ui(Brain + Helm), 제스처/모션은 GSAP를 채택한다.**

- **Brain (`@spartan-ng/brain`)** — npm으로 설치하는 헤드리스/접근성 레이어(Angular CDK 기반). 동작과 ARIA를 담당.
- **Helm (`@spartan-ng/helm`)** — CLI가 **프로젝트로 복사**하는 스타일 레이어(Tailwind v4 + class-variance-authority). 우리가 소유·수정한다.
- **GSAP** — spartan이 제공하지 않는 **끌어서 닫기·스와이프·속도 기반 던지기** 같은 제스처/모션을 담당.
- **Tailwind v4** — 이미 설치되어 있으며 Helm 프리셋(`hlm-tailwind-preset.css`)과 토큰 기반으로 사용.

### "외형 소유 / 동작 차용" 원칙과의 정합

spartan/ui는 이 방법론과 정확히 맞물린다. **동작·접근성은 Brain(차용)**, **외형은 Helm(프로젝트로 복사 → 소유)**. 완성형 프레임워크보다 제어권이 크고, 완전 자작보다 출발이 빠르다.

### 적응형 시트는 spartan + GSAP로 보강한다

spartan에는 `sheet`, `dialog` 컴포넌트가 있으나, 이 방법론의 핵심인 **화면 폭에 따라 바텀시트↔모달로 표현 자체가 갈리는 적응형 분기**와 **끌어서 닫기 제스처**는 기본 제공되지 않는다. 따라서:

- spartan `sheet`/`dialog`의 오버레이·포커스 트랩(Brain)을 **동작 기반**으로 차용하고,
- 화면별 분기는 **CDK `BreakpointObserver`** 로 판단하며,
- 끌어서 닫기·속도 기반 닫힘/되돌림은 **GSAP**로 구현하여,
- 이 모두를 **도메인을 모르는 공용 `Sheet` 컴포넌트** 안에 캡슐화한다(쓰는 쪽은 분기를 모른다).

## SSR 관련 주의사항 (이 프로젝트 고유)

UI 스택이 SSR과 안전하게 동작하기 위한 규칙. **이 주의점 자체가 좋은 시연 소재**다.

1. **브라우저 전용 코드는 플랫폼 분기로 감싼다.** `BreakpointObserver`, GSAP, 오버레이의 실제 DOM 측정 등은 서버에 `window`/`document`가 없으므로 `afterNextRender()` 또는 `isPlatformBrowser()` 가드 안에서만 실행한다. (리서치 §1, §2 참고)
2. **토스트/오버레이는 초기 SSR 마크업에 의존하지 않게.** sonner(toast)·동적 오버레이는 사용자 상호작용 이후 브라우저에서 뜨는 것이라 SSR HTML에 들어가지 않아야 한다.
3. **하이드레이션 불일치 방지.** 서버와 클라이언트가 같은 초기 트리를 그리도록, "모바일/데스크톱 분기 렌더"는 CSS(미디어 쿼리)로 우선 처리하고, JS 측정에 따른 구조 변경은 하이드레이션 이후로 미룬다.
4. **무거운 인터랙티브 위젯은 `@defer (hydrate on ...)`** 로 감싸 증분 하이드레이션과 결합한다(리서치 §2).

### SSR 가드의 캡슐화 위치 규칙

브라우저 체크(`isPlatformBrowser`/`afterNextRender`)는 **도메인 페이지에 흩뿌리지 않고**, 체크의 성격에 따라 아래 두 곳 중 하나에 캡슐화한다. 원칙은 *"쓰는 쪽(페이지)이 SSR을 의식하지 않게"* 이다.

| 브라우저 체크의 종류 | 캡슐화 위치 | 예 |
|----------------------|-------------|-----|
| **렌더/DOM 동작** — 요소 측정, 제스처, 애니메이션, 화면폭 분기 | **디자인 시스템 UI 컴포넌트** 안 (brain/helm을 감싼 우리 공용 컴포넌트) | 적응형 `Sheet`, 끌어서 닫기, 위치 계산 |
| **플랫폼 API 접근** — `window`, `localStorage`, `navigator`, `matchMedia` | **core의 플랫폼 provider/토큰** (UI 아님) | `WINDOW` 토큰, `StorageService`, 인증 인터셉터 |

추가 규칙:

- **Brain(CDK)이 이미 SSR-safe한 부분은 중복 가드하지 않는다.** CDK `Overlay`·`FocusTrap`·`BreakpointObserver`는 서버에서 안전하게 동작하므로, 우리가 직접 `afterNextRender`로 감쌀 표면은 주로 **우리가 추가하는 브라우저 전용 코드(GSAP 제스처)** 로 좁혀진다.
- **Helm(스타일 레이어)에는 브라우저 로직을 넣지 않는다.** Helm은 외형만. 동작 가드는 brain을 감싼 공용 컴포넌트 또는 core 서비스에 둔다.
- **페이지/템플릿에 `@if (isBrowser)` 흩뿌리기 금지** — 도메인 코드가 인프라를 알게 되고 하이드레이션 불일치를 유발한다.

## 결과 (Consequences)

### 긍정적

- 외형을 소유(Helm 복사)하므로 "spartan처럼 보임"을 피하고 디자인 정체성을 가져간다.
- Brain이 CDK 기반이라 Angular 22 SSR과 정합이 좋다.
- Tailwind v4가 이미 깔려 있어 추가 설정 비용이 작다.
- 데모 코드가 투명해 SSR/CSR 시연 포인트가 가려지지 않는다.

### 비용/리스크

- 적응형 시트·제스처는 **직접 조립**해야 한다(spartan이 끝까지 주지 않음). → 방법론을 이미 보유하므로 감당 가능.
- 의존성 추가: `@spartan-ng/brain`, `@spartan-ng/cli`, `@ng-icons`, `gsap`.
- GSAP·CDK Layout 등 브라우저 전용 코드의 SSR 가드를 빠짐없이 챙겨야 한다(위 주의사항).
