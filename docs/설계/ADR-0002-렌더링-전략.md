# ADR-0002 · 페이지별 렌더링 전략 (SSR / CSR / SSG)

> 상태: ✅ 결정됨
> 일자: 2026-06-22
> 관련: [`docs/기획/프로젝트.md`](../기획/프로젝트.md) §4, [`docs/기획/리서치.md`](../기획/리서치.md) §2

---

## 맥락 (Context)

이 프로젝트는 Angular 22의 하이브리드 렌더링을 시연한다. 핵심은 **세 가지 렌더링 모드를 한 도메인에서 보여주되, 각 페이지가 그 모드를 고른 이유가 명확한 것**이다.

선택 기준은 단 하나로 수렴한다 — **데이터가 얼마나 자주/누구에게 다르게 바뀌는가?**

| 성격 | 모드 | 근거 |
|------|------|------|
| 거의 안 바뀜(빌드 스냅샷으로 충분) | **Prerender(SSG)** | 가장 빠름, 서버 부하 0, 변경 시 재빌드 트레이드오프 |
| 자주 바뀜 / 요청마다 최신 + SEO 필요 | **Server(SSR)** | 요청마다 서버 렌더, Transfer State로 중복 호출 제거 |
| 사용자마다 다름(개인화·인증) | **Client(CSR)** | 브라우저에서만 렌더, SEO 불필요 |

---

## 결정 (Decision)

**랜딩(홈)만 SSG, 공개 콘텐츠는 SSR, 개인화/인증 영역은 CSR로 나눈다.**

| # | 경로 | 화면 | 렌더 모드 | 이유 / 시연 포인트 |
|---|------|------|-----------|---------------------|
| 1 | `/` | 홈/랜딩 | **Prerender (SSG)** | 콘텐츠가 거의 고정 → 빌드 타임 정적. 무거운 위젯은 `@defer (hydrate on viewport)`로 증분 하이드레이션 시연 |
| 2 | `/movies` | 영화 목록 | **Server (SSR)** | 정렬·동적 목록 + SEO. **Transfer State**로 하이드레이션 중복 호출 제거 시연 |
| 3 | `/movies/:id` | 영화 상세 | **Server (SSR)** | SEO 핵심 페이지. **SEO 메타 태그 + `httpResource` + Transfer State**의 교과서적 SSR 데모 |
| 4 | `/genre/:key` | 장르별 | **Server (SSR)** | 파라미터 라우트 SSR |
| 5 | `/search` | 검색 | **Server (SSR)** | 쿼리 파라미터(`q`) 기반 서버 렌더, 공유 가능한 URL |
| 6 | `/wishlist` | 위시리스트 | **Client (CSR)** | 인증 + 개인화. 쿠키 인증/SSR 인증 포워딩 논의와 연결(후속 ADR) |
| 7 | `/me/ratings` | 내 평점 | **Client (CSR)** | 개인화 |
| 8 | `/login` | 로그인 | **Client (CSR)** | 세션 발급 |

### 모드별 대표 시연 포인트

- **SSG** → 홈: 빌드 타임 정적 + `@defer(hydrate ...)` 증분 하이드레이션.
- **SSR** → 목록/상세/장르/검색: 서버 데이터 페치, Transfer State 중복 호출 제거, SEO 메타.
- **CSR** → 위시리스트/내 평점/로그인: 쿠키 인증, 개인화, 브라우저 전용 흐름.

---

## 검토했으나 채택하지 않은 것 (Rejected)

- **상세/장르를 Prerender(SSG) + `getPrerenderParams`로 미리 굽기**
  영화·장르가 적고 고정이라 `getPrerenderParams`(동적 라우트의 여러 인스턴스를 빌드 타임에 정적 생성) 시연 후보였다. 그러나 **구성 단순성·안정성**을 우선해 보류했다. 공개 콘텐츠는 일괄 SSR로 두어 "요청마다 최신 + SEO" 모델을 일관되게 보여준다.
  → 추후 SSG 심화 시연이 필요하면 별도 페이지/ADR로 다룬다.

---

## 결과 (Consequences)

### 긍정적
- 세 모드가 모두 등장하고, 각 모드의 선택 이유가 페이지 성격과 1:1로 대응해 학습 자료로 명확하다.
- 공개 콘텐츠를 SSR로 통일해 데이터 페칭·Transfer State 패턴을 한 가지 방식으로 반복 시연한다.

### 비용/리스크
- 홈 외에는 SSG가 없어, `getPrerenderParams` 같은 SSG 고급 기능은 이 단계에서 시연하지 않는다(의도된 범위 축소).
- SSR 페이지가 많아 SSR 서버가 항상 떠 있어야 한다(배포 시 Node 런타임 필요).

## 구현 매핑 (참고)

`src/app/app.routes.server.ts`에 다음 형태로 반영한다(상세 코드는 구현 단계에서):

```ts
export const serverRoutes: ServerRoute[] = [
  { path: '',            renderMode: RenderMode.Prerender }, // 홈
  { path: 'movies',      renderMode: RenderMode.Server },
  { path: 'movies/:id',  renderMode: RenderMode.Server },
  { path: 'genre/:key',  renderMode: RenderMode.Server },
  { path: 'search',      renderMode: RenderMode.Server },
  { path: 'wishlist',    renderMode: RenderMode.Client },
  { path: 'me/ratings',  renderMode: RenderMode.Client },
  { path: 'login',       renderMode: RenderMode.Client },
  { path: '**',          renderMode: RenderMode.Server },    // 폴백
];
```
