# ADR-0003 · 백엔드 / 데이터 계층 구조

> 상태: ✅ 결정됨
> 일자: 2026-06-22
> 관련: [`docs/기획/리서치.md`](../기획/리서치.md) §3, [ADR-0002](./ADR-0002-렌더링-전략.md)

---

## 맥락 (Context)

이 프로젝트는 SSR/CSR 혼합과 함께, 리서치 §3에서 다룬 **"SSR 서버↔백엔드 서버간 호출(CORS 없음, 내부 호스트), 브라우저↔백엔드 호출(같은 origin 프록시로 CORS 제거)"** 아키텍처를 실제로 시연하는 것이 목표다.

이를 제대로 보여주려면 백엔드를 **Angular SSR 서버와 분리된 별도 프로세스**로 두어야 한다. 단, "별도 프로세스"가 "별도 레포"를 의미하지는 않는다 — **하나의 레포 안에 Node 엔트리포인트를 2개** 두고 각각 다른 포트로 실행한다.

---

## 결정 (Decision)

**단일 레포 안에서 백엔드를 별도 Express 프로세스(포트 4001)로 분리하고, Angular SSR 서버(포트 4000)가 `/api`를 백엔드로 프록시한다. (리서치 B안)**

### 디렉토리 구조

```
angular-ssr-design/
├── src/
│   ├── server.ts          ← Angular SSR(4000) + '/api' 프록시 미들웨어
│   └── app/ ...            ← Angular 앱
├── backend/               ← 독립 목 백엔드(4001)
│   ├── index.ts           ← Express 엔트리, /api/* 라우트
│   ├── routes/            ← movies, auth, wishlist, ratings
│   └── data.ts            ← 인메모리 영화/장르/세션 데이터
└── package.json           ← 두 프로세스 실행 스크립트
```

### 호출 경로 (리서치 §3-5 아키텍처 재현)

```
[브라우저]   --상대경로 /api, 쿠키 자동-->  [SSR Express :4000] --프록시--> [백엔드 :4001]
                                                  ↑ 같은 origin → CORS 없음
[Angular SSR :4000] --http://localhost:4001/api, REQUEST 쿠키 포워딩--> [백엔드 :4001]
                                                  ↑ 서버간 내부호출 → CORS 없음
```

- **브라우저**는 항상 `/api`(상대경로)만 호출 → SSR Express가 받아 백엔드로 프록시 → **CORS 구조적 제거**.
- **Angular SSR**은 렌더 중 내부 호스트 `http://localhost:4001/api`로 **직접 서버간 호출** → CORS 없음, 내부 호스트 시연 성립.

### baseURL 분리

브라우저(`/api` 상대경로)와 서버(`http://localhost:4001/api` 절대경로)의 baseURL 차이는 **`API_BASE_URL` InjectionToken + 플랫폼별 provider**로 가른다(리서치 §3-2).

```ts
export const API_BASE_URL = new InjectionToken<string>('API_BASE_URL');
// app.config.ts(브라우저)      → { provide: API_BASE_URL, useValue: '' }                       // 상대경로
// app.config.server.ts(SSR)    → { provide: API_BASE_URL, useValue: 'http://localhost:4001' }   // 내부 호스트
```

> 환경별(dev/prod) 값은 환경 파일 또는 런타임 환경 변수로 주입한다(구현 단계에서 구체화).

### 실행 스크립트

```jsonc
// package.json scripts (구현 단계에서 확정)
"backend": "tsx watch backend/index.ts",                    // 백엔드 4001
"dev":     "concurrently \"npm:backend\" \"npm start\"",    // 백엔드 + SSR(4000) 동시
```

추가 의존성(예정): `concurrently`, `tsx`(또는 동등 TS 러너), `http-proxy-middleware`(SSR의 `/api` 프록시).

---

## 검토했으나 채택하지 않은 것 (Rejected)

- **A. 같은 Express에 `/api` 통합** — 셋업이 가장 단순하고 쿠키 인증도 쉽지만, SSR과 백엔드가 같은 프로세스라 "서버간 호출·내부 호스트" 개념이 흐려져 리서치 핵심을 제대로 시연하지 못한다.
- **프록시 없이 백엔드만 분리(브라우저가 4001 직접 호출)** — CORS를 "제거"가 아니라 "다루는"(허용 헤더 설정) 방향이라, 이 데모가 권장하는 "같은 origin 프록시" 모델과 어긋난다.

---

## 결과 (Consequences)

### 긍정적
- 리서치 §3의 서버간 호출·내부 호스트·CORS 제거·baseURL 분리·쿠키 포워딩을 **실제 동작으로** 보여줄 수 있다.
- 백엔드가 분리되어 있어 인증/데이터 페칭 후속 ADR의 논의가 현실적인 전제 위에서 진행된다.

### 비용/리스크
- 프로세스 2개 관리 → `concurrently`/`tsx` 등 dev 의존성과 실행 한 단계 추가.
- 배포 시 두 프로세스를 같은 호스트/컨테이너에 함께 올려야 한다.
- 프록시·내부 호스트 주소를 환경별로 관리해야 한다.
