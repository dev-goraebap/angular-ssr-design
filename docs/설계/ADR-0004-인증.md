# ADR-0004 · 인증 방식 (HttpOnly 쿠키 + SSR 포워딩)

> 상태: ✅ 결정됨
> 일자: 2026-06-22
> 관련: [`docs/기획/리서치.md`](../기획/리서치.md) §3-4, [ADR-0002](./ADR-0002-렌더링-전략.md), [ADR-0003](./ADR-0003-백엔드-데이터-계층.md)

---

## 맥락 (Context)

리서치 §3-4의 핵심 문제: **CSR은 모든 요청이 브라우저에서 출발해 토큰이 자동으로 따라가지만, SSR은 요청이 서버에서 출발해 브라우저 저장소(`localStorage`)에 접근할 수 없다.** 따라서 SSR 단계에서 `localStorage` 토큰에 의존하면 서버가 토큰을 못 읽어 비로그인 상태로 렌더된다(깜빡임·데이터 불일치).

이 프로젝트의 CSR 페이지(위시리스트·내 평점)는 인증이 필요하고, 백엔드는 별도 프로세스(ADR-0003)다. 인증을 **서버 호출과 클라 호출에서 한 줄기로 통일**하는 방식이 필요하다.

---

## 결정 (Decision)

**인증은 `HttpOnly` 세션 쿠키로 통일하고, SSR 렌더 시 `REQUEST`의 쿠키를 백엔드로 포워딩한다.**

### 핵심 구성

1. **로그인**: 브라우저(`/login`, CSR)에서 `POST /api/auth/login` → 백엔드가 **`Set-Cookie`(HttpOnly, Secure, SameSite)** 로 세션 쿠키 발급. SSR Express 프록시가 이 `Set-Cookie`를 브라우저로 그대로 전달.
2. **브라우저 호출**: 쿠키가 같은 origin(`:4000`)에 자동 전송되므로 별도 처리 불필요.
3. **SSR 호출**: HttpInterceptor에서 `isPlatformServer()`일 때 `REQUEST`의 `Cookie` 헤더를 읽어, 백엔드(`:4001`)로 나가는 요청에 `Cookie`로 **수동 포워딩**.
4. **인증 상태 판단**: `HttpOnly`라 JS가 토큰을 못 읽으므로, **`GET /api/me`** 로 현재 사용자를 확인해 `AuthService`(signal)에 보관한다.

### 인증 포워딩 인터셉터 (개념)

```ts
export const authForwardInterceptor: HttpInterceptorFn = (req, next) => {
  const platformId = inject(PLATFORM_ID);
  if (isPlatformServer(platformId)) {
    const request = inject(REQUEST);                 // @angular/core
    const cookie = request?.headers.get('cookie');
    if (cookie) {
      req = req.clone({ setHeaders: { cookie } });   // 사용자 쿠키를 백엔드로 전달
    }
  }
  // 브라우저: 쿠키 자동 전송 → 그대로 통과
  return next(req);
};
```

### 인증 상태 / 라우트 가드

- 앱 부팅 시(또는 가드에서) `GET /api/me` 호출 → 응답이 사용자면 로그인, 401이면 비로그인.
- CSR 보호 라우트(`/wishlist`, `/me/ratings`)는 **`CanActivate` 가드**로 `/api/me` 확인 → 미인증 시 `/login?returnUrl=...`로 보내고 로그인 후 복귀.
- 인증 상태는 `AuthService`의 signal로 보관하고, 헤더/내비의 로그인·로그아웃 표시에 사용.

### 쿠키 속성 (환경별)

| 속성 | 운영(prod) | 개발(dev, localhost) |
|------|-----------|----------------------|
| `HttpOnly` | ✅ | ✅ |
| `Secure` | ✅ (HTTPS) | ⚠️ 생략(HTTP localhost) |
| `SameSite` | `Lax`(같은 origin 프록시라 충분) | `Lax` |
| `Path` | `/` | `/` |

> 같은 origin 프록시(ADR-0003) 덕분에 크로스 사이트 쿠키 이슈가 없어 `SameSite=Lax`로 충분하다.

---

## 검토했으나 채택하지 않은 것 (Rejected)

- **localStorage 토큰** — SSR이 토큰을 못 읽어 비로그인 렌더 → 깜빡임/불일치. XSS에 토큰 노출. (SSR의 대표적 함정이라 문서로만 언급)
- **읽기 가능한(비 HttpOnly) 쿠키** — JS로 로그인 상태를 즉시 판단할 수 있어 간편하지만 XSS에 토큰이 노출된다. 보안을 우선해 보류.

---

## 결과 (Consequences)

### 긍정적
- 서버/클라 인증이 **쿠키 한 줄기로 통일**된다. 인터셉터 한 곳에서 플랫폼 분기만 처리.
- `HttpOnly`로 XSS에 토큰이 노출되지 않는다.
- `REQUEST` 포워딩, `/api/me` 기반 상태 판단 등 SSR 인증의 정석 패턴을 시연한다.

### 비용/리스크
- `HttpOnly`라 클라가 토큰을 직접 못 읽어, 인증 상태를 `/api/me`로 한 번 더 확인하는 단계가 필요.
- 로그인 직후 `Set-Cookie`가 프록시를 거쳐 전달되도록 SSR 프록시가 쿠키를 올바르게 패스해야 한다(구현 주의).
- 개발(HTTP)과 운영(HTTPS)의 `Secure` 속성 차이를 환경별로 관리해야 한다.

## 후속 작업 (Follow-up)

- [ ] 백엔드 인증 엔드포인트 스펙: `POST /api/auth/login`, `POST /api/auth/logout`, `GET /api/me`
- [ ] SSR 프록시(ADR-0003)가 `Set-Cookie`/`Cookie`를 양방향으로 올바르게 패스하는지 구현 시 검증
- [ ] `AuthService`(signal) + `CanActivate` 가드 + `authForwardInterceptor` 구현
- [ ] **데이터 페칭** 방식과의 연계: 인증 의존 응답은 Transfer State 캐시에서 제외(리서치 §3-3 데이터 누수 주의)
