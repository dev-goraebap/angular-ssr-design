# ADR-0005 · 데이터 페칭 (httpResource 우선 + HttpClient 쓰기)

> 상태: ✅ 결정됨
> 일자: 2026-06-22
> 관련: [`docs/기획/리서치.md`](../기획/리서치.md) §3-3 / §5, [ADR-0003](./ADR-0003-백엔드-데이터-계층.md), [ADR-0004](./ADR-0004-인증.md)

---

## 맥락 (Context)

SSR 페이지(목록·상세·장르·검색)는 서버에서 데이터를 가져와 HTML을 그리고, 브라우저 하이드레이션 시 **같은 API를 다시 호출하지 않아야** 한다(리서치 §3-3). Angular 22부터 `httpResource()`가 stable이 되었고 **Transfer State와 자동 통합**되어, 서버에서 미리 로드한 데이터가 브라우저 초기 렌더에서 그대로 재사용된다(리서치 §5).

---

## 결정 (Decision)

**읽기는 `httpResource()`(signal 기반, Transfer State 자동), 쓰기(변이)는 `HttpClient` 명령형 호출로 한다.**

| 용도 | 방식 | 예 |
|------|------|-----|
| **읽기(반응형)** | `httpResource()` | 영화 목록/상세, 장르별, 검색 결과 |
| **쓰기(변이)** | `HttpClient` (명령형) | 로그인/로그아웃, 위시리스트 담기/빼기, 별점 등록/수정/삭제 |

### 읽기 — httpResource

- 라우트 파라미터/쿼리(시그널)에 반응해 자동 재요청. 예: 상세 페이지의 `id` signal, 검색의 `q` signal.
- **Transfer State 자동**: SSR에서 페치한 결과가 직렬화되어 HTML에 실리고, 브라우저는 초기 렌더에서 재사용 → **중복 호출 제거**(리서치 §3-3).
- `loading`/`error`/`value` 상태를 signal로 제공 → 스켈레톤·에러 UI와 자연스럽게 연결(spartan `skeleton`/`empty`/`alert`).

### 쓰기 — HttpClient

- 변이는 사용자 액션에 대한 명령형 호출이라 `httpResource`(읽기 전용 반응형)와 맞지 않는다.
- 성공 후 관련 `httpResource`를 **`reload()`** 하거나 로컬 signal을 갱신해 화면을 반영.

### 공통 — baseURL / 인증

- baseURL은 `API_BASE_URL` 토큰 + 플랫폼별 provider로 가른다(ADR-0003). `httpResource`/`HttpClient` 모두 인터셉터를 거치므로 한 곳에서 처리.
- 인증 쿠키 포워딩도 인터셉터에서 처리(ADR-0004).

---

## 주의 — Transfer State와 인증/개인화 (리서치 §3-3)

- **인증 의존·개인화 응답은 Transfer State 캐시에서 제외**한다. 캐시 키가 인증/테넌트/로케일 헤더를 구분하지 못해 **다른 사용자 데이터가 섞일 수 있다.**
- 본 프로젝트에서 개인화 데이터(위시리스트·내 평점)는 **CSR 페이지**(ADR-0002)라 SSR Transfer State 경로를 타지 않으므로 위험이 구조적으로 낮다. 그래도 공개 데이터(영화 목록 등)와 개인화 데이터의 캐시 정책을 분리해 관리한다.
- 기본적으로 transfer cache는 `Authorization` 헤더 없는 `GET`/`HEAD`만 캐싱한다(리서치 §3-3).

---

## 검토했으나 채택하지 않은 것 (Rejected)

- **전부 `HttpClient` + 수동 `withHttpTransferCache()`** — Transfer State 내부 동작을 명시적으로 보여주는 장점이 있으나 코드량이 많다. Angular 22 기준 프로젝트의 권장 패턴은 `httpResource`이며, 자동 통합으로 "중복 호출 제거"를 더 적은 코드로 시연할 수 있어 보류.
- **읽기까지 `httpResource`로 통일하고 쓰기도 억지로 resource화** — 변이는 명령형이 자연스럽다. 혼용이 정답.

---

## 결과 (Consequences)

### 긍정적
- Angular 22 최신 권장 패턴(`httpResource` + 자동 Transfer State)을 시연.
- 읽기 코드가 간결하고, signal 기반이라 라우트 파라미터/검색어 변화에 선언적으로 반응.
- 중복 호출 제거가 적은 코드로 성립.

### 비용/리스크
- `httpResource`는 비교적 신규 API라 복잡한 페칭 시나리오(연쇄 요청·취소·재시도)는 추가 패턴이 필요할 수 있다.
- 읽기/쓰기 혼용 규칙을 팀이 일관되게 지켜야 한다(읽기=resource, 쓰기=HttpClient).
