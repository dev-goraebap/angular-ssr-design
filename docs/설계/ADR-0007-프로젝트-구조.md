# ADR-0007 · 프로젝트 구조 (FSD v2.1, 최소 레이어로 시작)

> 상태: ✅ 결정됨
> 일자: 2026-06-22
> 관련: [FSD 스킬], [ADR-0001](./ADR-0001-ui-기술스택.md), [ADR-0003](./ADR-0003-백엔드-데이터-계층.md), [ADR-0004](./ADR-0004-인증.md)

---

## 맥락 (Context)

디렉토리 구조로 **Feature-Sliced Design(FSD) v2.1**을 채택한다. 단, 8페이지 규모의 시연 프로젝트에 6개 레이어를 처음부터 다 펼치는 것은 과하다.

FSD 공식 원칙도 **"Start simple, extract when needed"** 와 **"Pages First"** 다 — 모든 코드는 일단 `pages/`에 두고, **실제로(가설이 아니라) 여러 곳에서 함께 쓰이고 경계가 안정될 때만** 하위 레이어로 추출한다.

---

## 결정 (Decision)

**`app` / `pages` / `shared` 3개 레이어로 시작하고, 여러 page에서 실제로 공유되는 것이 생기면 점진적으로 하위 레이어(`widgets`/`features`/`entities`)로 내려 추출한다. 비즈니스 무관 순수 UI는 먼저 spartan + GSAP로 `shared/ui`에 컴포넌트화한다.**

### 초기 구조 (Minimal FSD)

Angular의 부트스트랩 `app`과 FSD의 `app` 레이어는 의미가 일치(providers·routing·init)하므로 `src/app`을 FSD `app` 레이어로 그대로 사용한다.

```
src/
├── app/                      ← FSD app 레이어 = Angular 부트스트랩
│   ├── app.ts                   루트 컴포넌트
│   ├── app.config.ts            브라우저 providers (router, hydration, 인터셉터)
│   ├── app.config.server.ts     서버 providers (provideServerRendering)
│   ├── app.routes.ts            라우트
│   ├── app.routes.server.ts     렌더 모드 매핑 (ADR-0002)
│   └── layout/                  앱 셸·내비 (당장은 여기; 커지면 widgets로)
├── pages/                    ← 라우트별 페이지 (lazy loaded)
│   ├── home/                    SSG
│   ├── movies/                  SSR (무한 스크롤)
│   ├── movie-detail/            SSR
│   ├── genre/                   SSR
│   ├── search/                  SSR
│   ├── wishlist/                CSR (보호)
│   ├── ratings/                 CSR (보호)
│   └── login/                   CSR
├── shared/                   ← 비즈니스 로직 없는 인프라
│   ├── ui/                      spartan Helm 복사물 + 적응형 Sheet(GSAP) 등 순수 UI
│   ├── api/                     API 클라이언트, API_BASE_URL 토큰, 인터셉터, CRUD
│   ├── auth/                    세션/인증 유틸 (FSD 권장: auth는 shared/auth)
│   ├── lib/                     WINDOW 등 플랫폼 토큰, GSAP 유틸, breakpoint
│   └── config/                  환경 값, 라우트 상수
├── main.ts / main.server.ts / server.ts / styles.css
└── backend/                  ← FSD 범위 밖 (별도 프로세스, ADR-0003)
```

> `widgets/` `features/` `entities/` 폴더는 **지금 만들지 않는다**(빈 레이어 금지). 추출 시점에 생성한다.

### "순수 UI 먼저" — shared/ui 구축 순서

비즈니스를 모르는 순수 UI를 먼저 만든다(ADR-0001의 "도메인 모르는 공용 컴포넌트"가 곧 `shared/ui`):

1. spartan `init` → Helm 컴포넌트를 `shared/ui`로 복사(소유).
2. 필요한 spartan 컴포넌트 add: `button`, `card`, `input`, `badge`, `skeleton`, `dialog`, `sheet`, `avatar`, `tabs`, `empty`, `sonner` 등.
3. spartan이 안 주는 **적응형 `Sheet`(바텀시트↔모달) + 끌어서 닫기**를 GSAP·CDK `BreakpointObserver`로 조립해 `shared/ui/sheet`에 캡슐화(ADR-0001).
4. 페이지는 이 순수 UI를 조합하기만 한다.

### 점진적 추출 시나리오 (예시)

| 대상 | 시작 위치 | 추출 트리거 → 도착지 |
|------|-----------|----------------------|
| `MovieCard`, `Movie` 모델 | `pages/movies/` | 목록·상세·장르·검색·위시리스트 등 **여러 page에서 실제 사용** → `entities/movie/` |
| 위시리스트 담기 / 별점 남기기 | 해당 page | 상세·위시리스트 등 **2곳+에서 사용** → `features/add-to-wishlist`, `features/rate-movie` |
| 무한 스크롤 영화 리스트 블록 | `pages/movies/` | 장르·검색에서도 재사용 → `widgets/movie-list/` |
| 앱 내비게이션 | `app/layout/` | 복잡해지면 → `widgets/app-nav/` |

> 원칙(Golden Rule): **의심되면 `pages/`에 둔다.** 중복은 허용되며, 추출은 "지금 실제로 여러 곳에서 쓰이고 경계가 명확할 때"만.

## FSD 규칙 적용 (이 프로젝트에서 지킬 것)

- **Import 방향**: `app → pages → widgets → features → entities → shared`. 상위→하위만. 같은 레이어 슬라이스 간 교차 import 금지.
- **Public API**: 각 슬라이스는 `index.ts`(barrel)로만 외부 노출. `shared`는 세그먼트별 `index.ts`(`shared/ui/index.ts` 등).
- **도메인 기반 파일명**: `types.ts`/`utils.ts` 금지. `model/movie.ts`, `api/fetch-movies.ts`처럼.
- **shared에 비즈니스 로직 금지**: CRUD·API 클라이언트·UI 키트만. 도메인 규칙은 추출 시 `entities/`로.
- **인증 데이터는 `shared/auth`**: 세션/쿠키 유틸은 `user` 엔티티를 만들지 않고 `shared/auth`에 둔다(ADR-0004).
- **경로 alias**: `tsconfig`에 FSD alias(`@/shared/*` 등) 설정(구현 단계).

### SSR / Angular 통합 주의

- ADR-0001 캡슐화 규칙과 합치: **DOM 동작 가드는 `shared/ui` 컴포넌트 안**, **플랫폼 API 토큰은 `shared/lib`**.
- 서버/브라우저로 갈리는 provider(`API_BASE_URL` 등)는 `app/app.config.ts` / `app/app.config.server.ts`에서 주입(ADR-0003).
- 런타임 경계가 단일 `index.ts`로 안 되면 `index.server.ts` 같은 환경별 진입점 허용(FSD framework-integration 지침).

## 검토했으나 채택하지 않은 것 (Rejected)

- **6개 레이어 전체를 처음부터 구성** — 빈 레이어·조기 추출은 FSD 안티패턴(`insignificant-slice`). 최소 레이어로 시작한다.
- **단순 feature 구조(core/shared/features/pages)** — 가볍지만 FSD의 명시적 경계·의존성 규칙이라는 학습/유지보수 가치를 포기. FSD 채택이 확정이라 제외.
- **auth용 `user` 엔티티 생성** — 토큰/세션은 `shared/auth`가 정석(FSD 권장). 엔티티 조기 생성 회피.

---

## 결과 (Consequences)

### 긍정적
- 가볍게 시작(3레이어)하면서 FSD의 경계·의존성 규칙을 확보.
- "순수 UI(shared/ui) → 페이지 조합 → 공유되면 추출"이라는 진화 경로가 명확.
- Steiger 린터(`insignificant-slice`/`excessive-slicing`)로 구조를 검증 가능.

### 비용/리스크
- Public API(`index.ts`)와 import 방향 규칙을 일관되게 지켜야 한다(필요 시 ESLint 경계 규칙/Steiger 도입).
- "언제 추출할지" 판단이 사람에 따라 흔들릴 수 있어, 추출은 ADR/코멘트로 근거를 남긴다.
