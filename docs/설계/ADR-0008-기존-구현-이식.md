## 결정 (Decision)

**기존 코드를 본 프로젝트(`angular-ssr-design`)로 이식하면서, UI/UX 자산은 재활용하고 데이터·인증·SSR 계층은 ADR대로 교체/추가한다.**

### 재활용 (거의 그대로)

- **적응형 시트**(`shared/ui/adaptive-sheet`) — 데스크톱 모달 ↔ 모바일 바텀시트 스왑(ADR-0001과 합치).
- **URL 동기화 다이얼로그**(`features/open-movie`) — canonical URL 유지하며 모달 가로채기.
- **뒤로가기 처리** — CloseWatcher + `Location.subscribe`(popstate) + `NavigationStart` + `location.back()`.
- **무한 스크롤 + 스크롤 복원**(`pages/home`) — ADR-0006과 합치.
- **GSAP 제스처 셋업**, **breakpoint 서비스**, **테마**(`shared/lib`).
- **도메인 데이터**(SEED_MOVIES/SEED_GENRES) — 백엔드 시드 데이터로 이전.

### 교체 / 추가

- **데이터 계층**: Dexie(IndexedDB) → Express 백엔드(ADR-0003) + `httpResource`(ADR-0005). **가장 큰 작업.**
- **인증**: localStorage 세션 → HttpOnly 쿠키 + SSR 포워딩(ADR-0004).
- **SSR 인프라**: `@angular/ssr`/`platform-server`, `server.ts`/`main.server.ts`/`app.config.server.ts`/`app.routes.server.ts`(ADR-0002).
- **브라우저 API 가드**: GSAP/document/navigator/observer를 `afterNextRender`·플랫폼 토큰으로 격리(ADR-0001 캡슐화 규칙).

## URL 동기화 다이얼로그·뒤로가기는 SSR과 충돌하지 않는다 (핵심 검증)

가장 우려된 부분. 결론은 **충돌 없음, 오히려 상보적**. 근거:

**SSR은 최초 진입(서버 1회)만, 모달 가로채기는 그 이후 클라이언트 네비게이션만 담당한다.** 둘은 겹치지 않는다.

| 시나리오 | 렌더 주체 | 결과 |
|----------|-----------|------|
| 공유 링크로 `/movies/123` 직접 진입 | SSR | **canonical 페이지**로 렌더(모달 아님) — SSR이 잘하는 일 |
| `/` 진입 후 목록에서 영화 클릭 | 클라(하이드레이션 후) | `location.go()` + 모달 가로채기 |
| 모달에서 뒤로가기 | 클라 | `Location.subscribe`(popstate)로 모달 닫고 URL 복원 |

- `OpenMovieService.open()`은 **클릭 핸들러**에서만 호출 → 서버에서 실행되지 않음.
- `location.go/subscribe/back`은 Angular `Location`이라 SSR-safe.
- `CloseWatcher`는 이미 `globalThis` 런타임 체크 → SSR-safe.
- URL이 항상 **canonical**(`/movies/:id`)이라 공유·딥링크·SEO 보존 — 이 패턴의 설계 의도가 SSR과 정확히 맞물림.

→ **모달/뒤로가기 로직은 이식 시 거의 손대지 않는다.**

## 단계별 계획

1. **SSR 인프라 추가** — 본 레포는 이미 SSR 스캐폴드 보유. 라우트/렌더 모드 매핑(ADR-0002)부터.
2. **shared/ui·lib 이식** — 적응형 시트, GSAP, breakpoint, 테마 이식 + 브라우저 가드.
3. **백엔드 목 API 구축**(ADR-0003) — SEED 데이터를 `backend/`로 이전, 엔드포인트 작성.
4. **데이터 계층 전환** — Dexie 호출부를 `httpResource`/`HttpClient`(ADR-0005)로 교체.
5. **인증 쿠키화**(ADR-0004) — auth.service 재작성, SSR 포워딩 인터셉터, 가드.
6. **페이지 이식 + 렌더 모드 적용** — home/detail/genre/search(SSR), wishlist/ratings/login(CSR).
7. **검증** — 모달/뒤로가기/스크롤복원 하이드레이션 후 정상 + SSR HTML에 데이터 포함 + a11y.

## 결과 (Consequences)

### 긍정적
- 검증된 반응형 UX 자산(적응형 시트·URL 다이얼로그·뒤로가기·무한 스크롤)을 재활용해 구현 시간을 크게 단축.
- 스택이 동일해 이식 마찰이 작다.
- 모달/뒤로가기 패턴은 SSR과 충돌하지 않아 핵심 UX 리스크가 낮다.

### 비용/리스크
- **데이터 계층 전환**(IndexedDB→백엔드)이 가장 큰 작업이며, 화면 데이터 흐름 전반에 영향.
- localStorage→쿠키 전환으로 인증/테마 초기 상태 처리(하이드레이션 불일치) 점검 필요.
- 기존의 모듈레벨 GSAP import 등 SSR 비호환 패턴을 빠짐없이 가드해야 함.
